# /spec-kitty.domain

**Path reference rule:** When you mention directories or files, provide either the absolute path or a path relative to the project root (for example, `.kittify/memory/domains/<context>.md`). Never refer to a folder by name alone.

## 1. Purpose
This command creates or updates the single source of truth for a specific Bounded Context in `.kittify/memory/domains/<context>.md`.
It acts as a strict DDD consultant to help model the domain structure, terminology, and boundaries (explicitly avoiding feature-level or UI questions, which belong in `/spec-kitty.specify`).

## 2. User Input

```text
{{args}}
```

You **MUST** consider the user input before proceeding. If no input is provided, ask the user which domain they want to model.

## 3. Context Gathering & Initialization (Silent)
Before asking any questions, silently execute:
1. Ensure the directory `.kittify/memory/domains/` exists.
2. Read the global context (`README.md`, `constitution.md`).
3. Check if `.kittify/memory/domains/<context>.md` already exists. 
   - **If it exists:** Load it into memory. Your goal will be to **UPDATE** and refine this file rather than overwriting it from scratch. When updating, present the existing sections to the user and ask which areas they want to revisit or refine. Skip questions for sections the user confirms are still accurate.
   - **If it does not exist:** You will be creating a new file from scratch.
4. Scan other `.kittify/memory/domains/*.md` files to understand the broader ecosystem.

## 4. Discovery Proportionality & Early Exit
- **Complexity Assessment:** A tiny domain (e.g., just storing a static value) does not need all questions. Gauge the complexity from the user's initial input. If it's a simple CRUD concept, ask fewer questions.
- **Early Exit:** At ANY point during the interrogation, if the user explicitly says "done", "skip", "use defaults", "good enough", or "no more", you MUST immediately stop asking questions, summarize what you have, and proceed to the Confirmation step.

## 5. The Interrogation Loop
**CRITICAL:** You must ask EXACTLY ONE question at a time. Wait for the user's response before proceeding. 

These questions must strictly focus on DDD modeling (structure, state, language, boundaries) and MUST NOT overlap with feature-specification questions (like "Who is the user?" or "What are the edge cases?").

**CRITICAL BOUNDARY ENFORCEMENT:** If the user mentions UI elements (buttons, modals, pages, or DOM interactions), you MUST gently redirect them back to the Domain boundary. Tell them explicitly that UI details belong in the feature specification (using `/spec-kitty.specify`), not the Domain Context.

**Question 1: Ubiquitous Language & Core Concept**
- Question: "What are the core domain terms we need to define here? (e.g., 'In a Cart domain, a LineItem represents...')"
- Impact: Establishes vocabulary before modeling data.

**Question 2: Aggregate Root & Consistency**
- Question: "What is the Aggregate Root here, and what invariants (hard business rules) must ALWAYS be true within this boundary?"
- Example: "A cart cannot contain negative quantities."

**Question 3: Entities vs. Value Objects**
- Question: "Inside this boundary, what are the Entities (things with a unique ID that change over time) versus Value Objects (immutable things defined only by their attributes, like a Money or Address object)?"

**Question 4: Commands vs. Queries (Ports)**
- Question: "What Commands mutate the state of this domain, and what Queries just read from it?"
- *Note: This prevents mixing UI logic into domain logic.*

**Question 5: Lifecycle & State Transitions (Optional if trivial)**
- Question: "Does this Aggregate have a distinct lifecycle or state machine? (e.g., Active -> Checkout_Pending -> Completed?)"

**Question 6: Context Relationships**
- Question: "How does this context relate to other domains (e.g., Customer/Supplier, Shared Kernel, ACL)?"

**Question 7: Domain Events**
- Question: "What Domain Events does this context publish when its internal state changes?"

## 6. Summary & Confirmation
Once the interrogation is complete (or exited early):
1. Present a concise summary of the planned updates (or new file contents).
2. Explicitly ask: "Shall I write these updates to `.kittify/memory/domains/<context>.md`? (Yes/No/Edit)"

## 7. The Output Format (Write Phase)
Upon confirmation, write or update the file. Ensure the final file STRICTLY adheres to this template:

```markdown
# [Context Name] Bounded Context

> Auto-generated by spec-kitty domain command
> Created: [YYYY-MM-DD]
> Last Updated: [YYYY-MM-DD]

## 1. Ubiquitous Language
- **[Term]:** Definition...

## 2. Aggregate Root
**[Aggregate Name]**
- **Description:** ...
- **Invariants:**
  - [Rule 1...]

## 3. Structural Elements
### Entities
- `[EntityName]` (Identifier: `[ID Field]`): Description...

### Value Objects
- `[ValueObjectName]`: Description...

## 4. Lifecycle
- States: `[state1]` -> `[state2]`
- Transitions: `[Event]` triggers `[state1]` -> `[state2]`
*(Note: If the Aggregate has no complex lifecycle, just write `None` under the Lifecycle section rather than omitting it.)*

## 5. Ports (CQRS)
### Driving Ports (Inbound)
#### Commands (Mutations)
- `[CommandName]`: Action it performs...
#### Queries (Reads)
- `[QueryName]`: Data it returns...

### Driven Ports (Outbound)
- `[DependencyName]`: Abstract interface requirement...

## 6. Domain Events
- `[EventName]`: Triggered when...

## 7. Context Relationships
- **[Other Context]:** [Relationship Type (e.g., Upstream/ACL)] - [Brief Description]
```

## 8. Success Report
After writing the file:
1. Output the absolute path to the updated/created file.
2. Outline the sections modified.
3. Inform the user that Domain modeling is an optional planning step, and that they can continue refining it or (if they are ready to plan a feature using this domain) suggest running `/spec-kitty.specify` as a possible next step.
